1. Visión General

Este informe presenta el Análisis Exploratorio de Datos (EDA) exhaustivo realizado sobre el dataset Diabetes 130-US Hospitals for Years 1999-2008, obtenido del UCI Machine Learning Repository. Este análisis constituye la Fase 2 del proyecto de TFM, cuyo objetivo es desarrollar un framework de IA Generativa capaz de producir datos tabulares sintéticos que cumplan con los requisitos de anonimización bajo el Reglamento General de Protección de Datos (GDPR).

🎯 Objetivo del Análisis

El EDA tiene tres propósitos fundamentales:

  • Fidelidad de Datos: Identificar inconsistencias, valores atípicos y problemas de calidad.
  • Relación con el Target: Comprender qué variables influyen más en la readmisión hospitalaria.
  • Preparación para Modelos Generativos: Evaluar la cardinalidad y distribución de variables para CTGAN, TVAE y Tabular Diffusion.

1.1. Contexto del Dataset

El dataset contiene registros de encuentros clínicos de pacientes con diabetes en hospitales de Estados Unidos durante un período de 10 años (1999-2008). Cada registro representa un encuentro hospitalario de un paciente diagnosticado con diabetes, incluyendo información demográfica, procedimientos médicos, medicamentos administrados y el resultado del alta hospitalaria.

⚠️ Consideración GDPR

Este dataset contiene datos de salud, categorizados como "datos sensibles" bajo el Artículo 9 del GDPR. Cualquier uso de estos datos requiere garantías técnicas y organizativas apropiadas. La generación de datos sintéticos que preserven las propiedades estadísticas sin exponer información individual representa una solución para cumplir con el principio de minimización de datos.

2.2. Registros Inválidos: Fallecimientos y Hospice

Un hallazgo crítico del EDA fue la identificación de registros correspondientes a pacientes fallecidos o trasladados a cuidados paliativos (hospice). Estos registros representan falsos negativos para la variable objetivo readmitted, ya que un paciente fallecido no puede ser readmitido.

⚠️ Criterio de Exclusión

Se excluyeron 2.423 registros (2,38% del total) donde discharge_disposition_id toma los valores:

[11, 13, 14, 19, 20, 21]

Estos códigos corresponden a: fallecimiento en el hospital, traslado a hospice, o muerte en el lugar.

Adicionalmente, se eliminaron 3 registros con género 'Unknown/Invalid', resultando en un dataset final de 99.340 registros para el modelado.

2.3. Variables Eliminadas por Baja Varianza

El análisis de medicamentos reveló 7 compuestos con distribución extremadamente sesgada (más del 99% de los registros tienen valor 'No'), lo que las hace irrelevantes para el modelado generativo:

acetohexamide, troglitazone, examide, citoglipton, glimepiride-pioglitazone, metformin-rosiglitazone, metformin-pioglitazone

4. Análisis Bivariante: Relación con Readmisión

4.1. Variables Asociadas con Readmisión

El análisis bivariante reveló asociaciones importantes entre las variables predictoras y la readmisión hospitalaria:

🔍 Hallazgos Clave

  • number_inpatient: Los pacientes readmitidos tienen significativamente más visitas hospitalarias previas (media ~3.5 vs ~0.7 en no readmitidos). Esta es la variable predictora más fuerte identificada.
  • time_in_hospital: Los pacientes readmitidos tienden a tener estancias hospitalarias ligeramente más largas en el encuentro índice.
  • number_diagnoses: Mayor número de diagnósticos se asocia con mayor probabilidad de readmisión.
  • Insulina: Los pacientes en tratamiento con insulina muestran tasas de readmisión más altas.

4.2. Correlaciones entre Variables Numéricas

La matriz de correlación identificó multicolinealidad esperada entre variables de utilización:

  • number_inpatient correlaciona positivamente con num_medications (r ≈ 0.35) y number_diagnoses (r ≈ 0.28)
  • time_in_hospital correlaciona con num_procedures (r ≈ 0.30)
  • num_lab_procedures correlaciona con number_diagnoses (r ≈ 0.40)

📊 Implicación para Privacidad

Las correlaciones fuertes entre variables representan un riesgo de inferencia: un atacante podría deducir valores sensibles (ej. número de diagnósticos) a partir de variables aparentemente inocuas (ej. número de procedimientos de laboratorio). Los modelos generativos deben preservar estas correlaciones para utilidad estadística, pero el dataset sintético resultante debe ser evaluado con ataques de inferencia de membresía (MIA) para garantizar que no permiten reconstrucción de individuos específicos.

5. Transformaciones e Ingeniería de Variables

5.1. Agrupación de Diagnósticos ICD-9

Una de las transformaciones más significativas fue la consolidación de cientos de códigos ICD-9 en 9 categorías clínicas manejables. Esta reducción de cardinalidad es esencial para que los modelos generativos aprendan distribuciones significativas.

🎯 Beneficio para Modelos Generativos

Reducir la cardinalidad de diag_1 de ~700 códigos únicos a 9 categorías mejora drásticamente la capacidad del modelo de aprender patrones significativos. Además, esta transformación tiene un beneficio colateral de privacidad: los diagnósticos específicos son información sensible; agruparlos reduce el riesgo de re-identificación.

5.2. Variables Creadas (Feature Engineering)

prior_visits

Variable sintética que resume la historia de utilización hospitalaria:

prior_visits = number_outpatient + number_emergency + number_inpatient

  • Media: 1.2 visitas
  • Mediana: 0 visitas (60% sin historial)
  • Máximo: 80 visitas (outlier extremo)

any_med_change

Variable binaria que indica si hubo modificación en la medicación durante el encuentro:

any_med_change = 1 si change == 'Ch', else 0

  • Proporción: ~18% de los encuentros tuvieron cambio de medicación

6. Conclusiones e Implicaciones para el Modelado

6.1. Resumen de Hallazgos

  1. Calidad de Datos: El dataset original requería limpieza significativa, particularmente en variables con nulos extremos (>80%) y registros de pacientes fallecidos que sesgaban el target.
  2. Desequilibrio de Clases: El 11.1% de readmisiones justifica el uso de modelos generativos condicionales para balancear clases durante la síntesis.
  3. Cardinalidad Reducida: La agrupación ICD-9 de diagnósticos mejora la viabilidad del modelado y reduce riesgos de privacidad.
  4. Correlaciones Preservadas: Las relaciones entre variables de utilización deben ser capturadas fielmente por los modelos generativos.

6.2. Recomendaciones para la Fase 3 (Modelado Generativo)

🤖 Selección de Modelos

  • CTGAN: Recomendado para manejar el desequilibrio de clases mediante muestreo condicional en la variable readmitted.
  • TVAE: Útil para generación rápida de grandes volúmenes de datos sintéticos; puede "suavizar" distribuciones multimodales.
  • TabDDPM: Estado del arte para capturar correlaciones complejas; recomendado para validación final de fidelidad estadística.

6.3. Consideraciones de Privacidad y GDPR

🔒 Riesgos Identificados

  • Re-identificación: Combinaciones únicas de variables (ej. edad + diagnóstico + especialidad médica) podrían permitir vinculación con registros externos.
  • Inferencia: Las correlaciones fuertes permiten deducir valores no observados.
  • Membresía: Un atacante podría determinar si un individuo específico estaba en el dataset de entrenamiento.

🛡️ Próximas Validaciones (Fase 5)

  • Distance to Closest Record (DCR): Demostrar que los datos sintéticos no son copias exactas de registros reales.
  • Membership Inference Attack (MIA): Simular ataques para verificar que el modelo no memorizó individuos específicos.
  • Validación TSTR: Entrenar modelos predictivos con datos sintéticos y testear con reales para cuantificar utilidad.

6.4. Conclusión Final

El EDA ha establecido una base sólida para la generación de datos sintéticos. Las transformaciones aplicadas (exclusión de fallecidos, agrupación ICD-9, imputación de nulos, ingeniería de variables) han producido un dataset limpio y estructurado que mantiene las propiedades estadísticas esenciales mientras reduce riesgos de privacidad.

El siguiente paso es implementar los modelos generativos (CTGAN, TVAE, TabDDPM) y evaluar su capacidad para producir datos que:

  1. Sean estadísticamente indistinguibles de los reales (fidelidad)
  2. Preserven la utilidad predictiva (TSTR validation)
  3. Resistan ataques de re-identificación e inferencia (privacidad)
  4. Crucen el umbral legal de "datos anónimos" bajo GDPR
